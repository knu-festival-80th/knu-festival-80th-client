# 롤링페이퍼 배치 미리보기 위치 불일치 문제

## 상황

롤링페이퍼 작성 플로우는 다음 순서로 동작한다.

1. 사용자가 메시지를 작성한다.
2. 포스트잇 디자인을 선택한다.
3. 배치도 화면에서 포스트잇을 붙일 위치를 선택한다.
4. `롤링페이퍼 붙이기` 버튼을 누르면 선택 좌표를 서버에 전달한다.

문제는 3번 배치 단계에서 발생했다.

사용자가 큰 배치도에서 포스트잇을 호반우 주변에 올려둔 것처럼 보였지만, 하단 우측 미니맵의 빨간 점은 다른 위치를 가리켰다. 실제로 붙이기를 하면 큰 배치도에서 보이던 위치가 아니라 미니맵이 표시한 위치에 포스트잇이 저장되었다.

즉, 서버에 전달되는 좌표와 미니맵 좌표는 일치했지만, 큰 배치도에서 보이는 새 포스트잇 미리보기만 다른 위치에 렌더링되는 문제가 있었다.

## 재현 조건

빈 보드에서는 클릭 위치와 미리보기 위치가 정상적으로 일치했다.

하지만 기존 포스트잇이 여러 개 존재하는 보드에서는 다음 현상이 나타났다.

- 큰 배치도에서 보이는 새 포스트잇 위치와 미니맵의 빨간 점 위치가 달랐다.
- 기존 포스트잇 개수가 늘어날수록 큰 배치도에서 보이는 새 포스트잇 위치가 더 어긋났다.
- 실제 저장 결과는 미니맵의 빨간 점 위치와 일치했다.

이 조건 때문에 처음에는 포인터 좌표 변환이나 미니맵 클릭 처리 문제처럼 보였다.

## 처음 의심한 원인

초기에는 `clientX`, `clientY`를 보드 좌표로 변환하는 과정이 잘못됐을 가능성을 먼저 의심했다.

배치 미리보기에서는 화면 좌표를 다음 순서로 변환한다.

1. 브라우저 포인터 좌표를 가져온다.
2. 미리보기 보드의 위치와 크기를 기준으로 보드 내부 좌표로 변환한다.
3. 캔버스 기준 좌표를 0~100 상대 좌표로 변환한다.
4. 해당 좌표를 미니맵 점, 큰 배치도 미리보기, 서버 요청 payload에 사용한다.

또한 미니맵 위를 클릭했을 때 투명 입력 레이어가 이벤트를 가로채는 문제도 가능성이 있었다. 그래서 입력 레이어를 분리하고, 미니맵보다 낮은 z-index를 사용하도록 조정했다.

하지만 이 수정만으로는 문제가 해결되지 않았다. 미니맵과 서버 좌표는 계속 일치했고, 큰 배치도에서 보이는 포스트잇만 어긋났기 때문에 핵심 원인은 좌표 계산이 아니라 렌더링 방식에 있었다.

## 실제 원인

실제 원인은 `RollingPaperSticker` 컴포넌트의 기본 position 클래스와 배치 미리보기에서 넘겨주는 position 클래스가 충돌한 것이었다.

`RollingPaperSticker`는 내부 텍스트를 스티커 위에 올리기 위해 루트 요소에 기본적으로 `relative` 클래스를 붙이고 있었다.

```tsx
<div className={`relative [container-type:inline-size] ${className}`} style={style}>
  ...
</div>
```

그런데 배치 미리보기에서는 포스트잇을 보드 위의 특정 좌표에 배치해야 하므로 외부에서 `absolute` 클래스를 넘기고 있었다.

```tsx
<RollingPaperSticker
  className="absolute z-30"
  style={{
    left: `${selectedPlacement.x}%`,
    top: `${selectedPlacement.y}%`,
    transform: 'translate(-50%, -50%)',
  }}
/>
```

결과적으로 하나의 요소에 `relative`와 `absolute`가 동시에 들어갔다.

Tailwind CSS에서는 클래스 문자열에서 뒤에 오는 클래스가 항상 이기는 구조가 아니다. 최종 CSS 적용 순서는 Tailwind가 생성한 CSS 순서에 영향을 받는다. 따라서 `className` 문자열 안에서 `absolute`가 뒤에 있어도, 실제 CSS에서는 `relative`가 `absolute`를 덮을 수 있다.

이 경우 포스트잇은 의도한 absolute 좌표에 배치되지 않고 일반 문서 흐름에 참여한다.

그래서 기존 포스트잇이 없을 때는 문제가 잘 보이지 않았지만, 기존 포스트잇이 여러 개 생기면 새 포스트잇 미리보기가 기존 요소들의 흐름에 밀려서 보였다.

## 왜 미니맵은 맞았는가

미니맵은 DOM 흐름에 영향을 받지 않고 `selectedPlacement.x`, `selectedPlacement.y` 값을 직접 점 위치로 렌더링한다.

```tsx
<span
  style={{
    left: `${selectedPlacement.x}%`,
    top: `${selectedPlacement.y}%`,
    transform: 'translate(-50%, -50%)',
  }}
/>
```

서버 요청 payload도 같은 `selectedPlacement` 값을 사용한다.

따라서 미니맵과 서버 저장 좌표는 정상적으로 일치했다.

문제는 큰 배치도에서 새 포스트잇 컴포넌트가 `absolute`로 배치되지 못하고 `relative`로 렌더링되면서 시각적 위치만 틀어진 것이었다.

## 해결 방법

`RollingPaperSticker`가 외부에서 이미 position 클래스를 받은 경우에는 기본 `relative`를 붙이지 않도록 수정했다.

```tsx
function hasPositionClass(className: string) {
  return /\b(absolute|fixed|relative|sticky)\b/.test(className);
}

const positionClassName = hasPositionClass(className) ? '' : 'relative';

return (
  <div className={`${positionClassName} [container-type:inline-size] ${className}`} style={style}>
    ...
  </div>
);
```

이 방식의 의도는 다음과 같다.

- 일반 사용처에서는 위치 클래스가 없으므로 기존처럼 `relative`를 유지한다.
- 스티커 내부 텍스트 오버레이는 계속 정상적으로 동작한다.
- 배치 미리보기처럼 외부에서 `absolute`를 명시하는 경우에는 기본 `relative`를 제거한다.
- 큰 배치도 미리보기, 미니맵, 서버 요청 좌표가 모두 같은 `selectedPlacement` 기준으로 일치한다.

## 함께 적용한 보완

배치 미리보기의 포인터 입력도 별도 투명 레이어로 분리했다.

기존 스티커나 SVG가 직접 포인터 이벤트 타깃이 되면, 클릭 대상이 달라지면서 좌표 계산이 흔들릴 가능성이 있다. 이를 막기 위해 보드 위에 투명 입력 레이어를 두고, 모든 배치 입력은 해당 레이어에서만 처리하도록 했다.

또한 미니맵은 안내 UI이므로, 투명 입력 레이어보다 높은 z-index를 유지하게 했다. 이로써 미니맵 영역 클릭이 배치 좌표로 잘못 처리되는 것을 방지했다.

## 결과

수정 후에는 다음 기준이 일치한다.

- 큰 배치도에서 보이는 새 포스트잇 위치
- 하단 우측 미니맵의 빨간 점 위치
- `롤링페이퍼 붙이기` 요청에 포함되는 서버 전달 좌표
- 실제 보드에 저장된 후 렌더링되는 포스트잇 위치

## 재발 방지 체크포인트

스티커 컴포넌트를 재사용할 때는 루트 요소에 기본 position 클래스를 고정하지 않아야 한다.

특히 다음처럼 공용 컴포넌트가 내부에서 position을 강제하고, 외부에서도 position을 넘기는 구조는 피해야 한다.

```tsx
<div className={`relative ${className}`} />
```

대신 외부 배치가 필요한 컴포넌트는 다음 중 하나를 선택해야 한다.

- position 클래스를 props로 명시적으로 제어한다.
- 외부에서 position 클래스가 들어온 경우 기본 position을 생략한다.
- 루트 wrapper와 내부 content wrapper를 분리해서 외부 배치와 내부 오버레이 기준을 분리한다.

이번 케이스에서는 기존 사용처 영향이 가장 적은 두 번째 방식을 선택했다.
