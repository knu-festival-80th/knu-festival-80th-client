# 롤링페이퍼 배치 검증 정리

## 문제

롤링페이퍼 작성 과정에서 새 포스트잇을 기존 포스트잇과 겹치는 위치에 올려도 `롤링페이퍼 붙이기` 버튼이 활성화되는 문제가 있었다.

사용자 입장에서는 화면상으로 명확히 겹쳐 보이는데도 제출 버튼이 활성화되므로, 프론트가 배치 가능 여부를 제대로 확인하지 않는 것처럼 보인다. 이후 백엔드에서 충돌을 거부하더라도 사용자는 이미 가능한 위치라고 인지한 상태이기 때문에 UX가 어색해진다.

## 원인

작성 모달의 배치 단계에서 `findNearestAvailableRollingPaperPlacement`를 사용하고 있었다.

이 함수는 요청한 위치가 이미 사용 중이면 주변을 탐색해서 가장 가까운 빈 위치를 찾아주는 함수다. 따라서 사용자가 실제로 찍은 위치가 겹치더라도, 내부적으로는 다른 안전한 위치가 계산되어 `selectedPlacement`가 존재하게 된다.

그 결과 버튼 활성화 조건은 아래처럼 동작했다.

1. 사용자가 겹치는 위치를 선택한다.
2. 프론트가 가까운 빈 위치를 자동 탐색한다.
3. `selectedPlacement`가 `null`이 아니므로 버튼이 활성화된다.
4. 사용자는 현재 보이는 위치에 붙일 수 있다고 오해한다.

즉 문제의 핵심은 **배치 가능 여부 검증**과 **자동 위치 보정**이 같은 단계에서 섞여 있었다는 점이다.

## 해결

작성 모달에서는 자동 위치 보정을 사용하지 않고, 사용자가 선택한 위치 자체를 기준으로 검증하도록 변경했다.

현재 흐름은 아래와 같다.

1. 사용자가 보드 위에서 포스트잇 위치를 선택한다.
2. 선택 좌표를 보드 경계 안으로만 clamp한다.
3. `isRollingPaperPlacementAvailable`로 기존 포스트잇 및 중앙 금지 영역과의 충돌을 검사한다.
4. 충돌하면 `롤링페이퍼 붙이기` 버튼을 비활성화한다.
5. 안내 문구로 `이미 붙은 포스트잇과 겹쳐요. 빈 위치로 옮겨주세요`를 보여준다.
6. 제출 직전에도 같은 검증을 다시 통과해야 백엔드 요청을 보낸다.

## 변경된 정책

기존 기본 충돌 정책은 유지한다.

```ts
ROLLING_PAPER_COLLISION_SCALE = 0.6;
```

이 값은 서비스 전체 배치 밀도와 백엔드 협의 기준에 사용된다. 100개 수용을 고려해 스티커 전체 외곽이 아니라 핵심 충돌 영역 기준으로 판단하는 값이다.

프론트의 작성 단계도 동일한 충돌 기준을 사용한다.

```ts
ROLLING_PAPER_CLIENT_COLLISION_SCALE = ROLLING_PAPER_COLLISION_SCALE;
```

사용자가 직접 배치하는 순간에도 기존 보드 밀도 정책과 같은 기준으로 적당한 겹침을 허용한다. 프론트와 백엔드가 같은 충돌 영역 기준을 공유해야, 프론트에서는 가능하다고 보였는데 서버에서 거절되는 상황을 줄일 수 있다.

## 백엔드와의 역할 분리

프론트 검증은 사용자 경험과 불필요한 요청 감소를 위한 1차 방어선이다.

하지만 최종 저장 가능 여부는 백엔드가 반드시 다시 검증해야 한다. 여러 사용자가 동시에 같은 보드에 작성할 수 있기 때문에, 프론트에서 가능한 위치였더라도 제출 순간에는 이미 다른 사용자가 같은 위치를 차지했을 수 있다.

권장 흐름은 아래와 같다.

1. 프론트가 현재 보드의 포스트잇 목록을 기준으로 1차 충돌 검증을 한다.
2. 통과한 경우에만 `POST /canvas/postits` 요청을 보낸다.
3. 백엔드는 최신 DB 상태 기준으로 다시 충돌 검증을 한다.
4. 충돌이 없으면 저장한다.
5. 충돌이 있으면 저장을 거부하고 프론트는 최신 포스트잇 목록을 다시 불러온다.

## 기대 효과

- 사용자가 겹치는 위치에 포스트잇을 붙일 수 있다고 오해하는 상황을 줄인다.
- 백엔드로 보내는 불필요한 충돌 요청을 줄인다.
- 프론트와 백엔드가 각각 1차 검증과 최종 검증을 맡아 동시성 상황에서도 안정성을 확보한다.
