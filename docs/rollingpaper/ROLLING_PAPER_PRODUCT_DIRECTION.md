# 롤링페이퍼 기획 및 구현 방향 재정의

## 배경

롤링페이퍼 기능을 구현하면서 처음에는 “사용자가 보드 위에 자유롭게 포스트잇을 붙인다”는 관점으로 접근했다.

하지만 예상 참여 규모와 실제 사용자 흐름을 다시 생각해보면, 단순히 보드를 여러 장 늘리는 방식만으로는 UX와 운영 측면에서 한계가 있다.

## 문제 정의

### 사용자는 왜 롤링페이퍼를 작성할까?

롤링페이퍼는 단순한 게시판보다 감정적이고 참여적인 기능이다.

사용자는 축제에 대한 감상, 축하 메시지, 기억에 남는 순간, 함께한 경험을 남기고 싶어서 롤링페이퍼를 작성한다. 따라서 메시지를 작성하는 행위 자체에 어느 정도의 의미와 맥락이 있어야 한다.

아무 주제 없이 “메시지를 남겨주세요”만 제공하면 사용자는 무엇을 써야 할지 고민하게 되고, 참여 몰입도도 낮아질 수 있다.

### 예상 참여 규모 문제

최대 예상 참여 인원을 `8,000명`으로 보면, 한 사람이 하나씩만 작성해도 포스트잇은 `8,000개`가 필요하다.

현재 한 보드당 최대 수용량을 `100개`로 잡으면 필요한 보드는 다음과 같다.

```txt
8,000명 / 보드당 100개 = 80개 보드
```

즉 단일 롤링페이퍼 흐름에서 보드를 80장 이상 운영해야 한다.

이 경우 사용자는 롤링페이퍼 페이지에 들어와 간단한 방명록을 남기려는 상황에서도, 여러 보드 중 어디에 작성해야 하는지 고민해야 한다. 또한 보드 이동, 빈 공간 탐색, 가득 찬 보드 처리 등이 UX 부담으로 이어질 수 있다.

### 동시성 및 보드 포화 문제

한 보드가 거의 가득 찬 상태에서 여러 사용자가 동시에 포스트잇을 작성할 수 있다.

프론트에서는 배치 가능해 보였던 위치도 submit 순간에는 다른 사용자가 먼저 차지했을 수 있다. 또는 작성 시점에는 여유가 있었지만 저장 시점에는 보드가 가득 찰 수 있다.

이 경우 사용자에게 아래와 같은 안내를 제공해야 한다.

```txt
이 보드는 방금 가득 찼어요. 다음 보드로 이동하시겠어요?
```

하지만 방명록을 간단히 남기려는 사용자 입장에서는 이 흐름 자체가 부담스럽다. 메시지를 작성한 뒤 다시 다른 보드로 이동하거나 위치를 다시 잡아야 한다면 이탈 가능성이 높아진다.

## 대안 1: 거대한 지도형 롤링페이퍼

### 아이디어

롤링페이퍼를 단순 보드가 아니라 하나의 큰 지도처럼 만든다.

보드 안에 축제와 관련된 랜드마크를 배치하고, 사용자는 지도를 탐색하듯이 이동하면서 특정 랜드마크 주변에 롤링페이퍼를 붙인다.

이 방식은 롤링페이퍼에 게임적인 탐색 경험을 부여할 수 있다.

예시:

```txt
대형 축제 지도
-> 랜드마크 탐색
-> 원하는 장소 주변에 포스트잇 작성
-> 사용자가 직접 위치에 의미를 부여
```

### 장점

- 롤링페이퍼 작성 행위에 공간적 의미를 부여할 수 있다.
- 사용자가 축제 지도를 탐색하는 것 같은 게이미피케이션 경험을 줄 수 있다.
- 단순 게시판보다 시각적으로 더 강한 인상을 줄 수 있다.

### 문제점

- 보드가 커질수록 렌더링 부담이 커진다.
- 모바일에서 확대/축소, 팬 이동, 수많은 스티커 렌더링을 안정적으로 처리해야 한다.
- 사용자가 메시지를 간단히 남기려는 상황에서는 탐색 시간이 길어질 수 있다.
- “어디에 붙여야 하지?”라는 고민이 늘어나 작성 진입 장벽이 높아질 수 있다.
- 구현 난이도가 높고, 성능 최적화와 좌표 관리가 복잡해진다.

### 판단

지도형 롤링페이퍼는 의미 부여와 시각적 재미는 크지만, 현재 서비스의 핵심 목표인 “축제 참여자가 간단하게 메시지를 남기는 경험”과는 다소 멀어질 수 있다.

특히 모바일 중심 서비스에서 사용자가 긴 탐색 과정을 거쳐야 한다면 오히려 불편함이 커질 가능성이 있다.

따라서 1차 구현 방향으로는 적합하지 않다.

## 대안 2: 질문 카테고리 기반 롤링페이퍼

### 아이디어

롤링페이퍼를 하나의 거대한 보드로 운영하지 않고, 사용자가 답변하기 쉬운 질문 카테고리로 나눈다.

각 카테고리는 하나의 주제를 가진다.

예시:

```txt
오늘의 기분을 작성해보세요
가장 맛있었던 주막 음식은?
가장 재밌었던 공연은?
오늘 가장 기억에 남는 순간은?
경북대학교 80주년에 남기고 싶은 말
```

각 카테고리 안에는 여러 개의 채널 또는 보드를 미리 만들어둔다.

예시:

```txt
카테고리 5개
카테고리당 보드 20개
보드당 포스트잇 100개

5 * 20 * 100 = 10,000개 수용 가능
```

### 장점

- 사용자가 무엇을 써야 할지 바로 이해할 수 있다.
- 질문 자체가 작성 동기를 제공한다.
- 전체 수용량을 자연스럽게 분산할 수 있다.
- 특정 보드에 사용자가 몰리는 문제를 줄일 수 있다.
- 보드가 가득 찼을 때 “다음 보드로 이동하시겠어요?” 같은 UX 부담을 줄일 수 있다.
- 백엔드도 카테고리와 보드 단위로 데이터를 나누어 관리할 수 있다.

### 사용자 흐름

```txt
롤링페이퍼 페이지 진입
-> 질문 카테고리 선택
-> 해당 카테고리의 여유 있는 보드 진입
-> 메시지 작성
-> 포스트잇 디자인 선택
-> 위치 선택
-> 저장
```

또는 더 단순하게는 카테고리를 먼저 보여주고, 사용자가 카테고리를 선택하면 프론트/백엔드가 자동으로 여유 있는 보드를 배정할 수 있다.

```txt
질문 선택
-> 여유 있는 보드 자동 배정
-> 작성 및 배치
```

### 수용량 계산

보드당 최대 포스트잇 수를 `100개`로 잡는 경우:

```txt
카테고리 5개
카테고리당 보드 20개
보드당 100개
= 10,000개
```

예상 참여자 `8,000명` 기준으로도 여유가 있다.

```txt
10,000개 수용 가능 - 8,000명 예상 참여 = 2,000개 여유
```

### 동시성 문제 완화

카테고리별로 여러 보드를 미리 만들어두면 특정 보드 하나에 모든 사용자가 몰리지 않는다.

또한 사용자가 직접 꽉 찬 보드에 들어가는 상황을 줄이고, 백엔드가 여유 있는 보드를 우선 배정할 수 있다.

예시 정책:

```txt
카테고리 선택
-> 백엔드가 해당 카테고리에서 가장 여유 있는 보드 반환
-> 사용자는 해당 보드에 작성
```

또는:

```txt
카테고리 선택
-> 프론트가 보드 리스트와 잔여 개수 조회
-> 여유 있는 보드를 기본 선택
```

이렇게 하면 submit 순간에 보드가 가득 차서 다시 작성해야 하는 상황을 줄일 수 있다.

## 최종 방향

현재 기획과 구현 우선순위를 고려하면, 두 번째 아이디어인 **질문 카테고리 기반 롤링페이퍼**가 더 적합하다.

지도형 롤링페이퍼는 매력적인 아이디어지만, 모바일 UX와 구현 난이도 측면에서 리스크가 크다.

반면 질문 카테고리 기반 방식은 사용자가 메시지를 작성할 이유를 명확하게 제공하고, 대량 참여 수용 문제도 구조적으로 해결할 수 있다.

## 권장 구조

### 도메인 구조

```ts
type RollingPaperCategory = {
  id: number;
  title: string;
  description?: string;
  boardCount: number;
};

type RollingPaperBoard = {
  id: number;
  categoryId: number;
  boardVariant: number;
  noteCount: number;
  maxNoteCount: number;
};

type RollingPaperNote = {
  id: number;
  boardId: number;
  colorId: 'red' | 'yellow' | 'green' | 'blue' | 'purple' | 'pink';
  message: string;
  x: number;
  y: number;
  createdAt: string;
};
```

const rollingPaperCategories = [

## 백엔드 구현 방향

백엔드는 카테고리별로 여러 보드를 미리 생성해둘 수 있다.

예시:

```txt
카테고리 5개
각 카테고리당 보드 20개
총 100개 보드 사전 생성
```

사용자가 카테고리를 선택하면 백엔드는 해당 카테고리에서 여유 있는 보드를 반환한다.

```ts
type GetAvailableBoardResponse = {
  categoryId: number;
  boardId: number;
  boardVariant: number;
  noteCount: number;
  maxNoteCount: number;
};
```

포스트잇 저장 시에는 기존과 동일하게 최종 검증을 백엔드에서 수행한다.

```txt
1. boardId 기준 보드 조회
2. 보드 최대 개수 확인
3. 포스트잇 좌표 검증
4. 중앙 금지 영역 검증
5. 기존 포스트잇과 충돌 검증
6. 저장
```

## 프론트 구현 방향

프론트는 기존 단일 보드 흐름을 다음 구조로 확장한다.

```txt
롤링페이퍼 인트로
-> 메시지 남기러 가기
-> 카테고리 선택 페이지
-> 여유 있는 보드 진입
-> 작성 모달
-> 위치 선택
-> 저장 완료
```

기존 보드 컴포넌트는 `boardId`와 `categoryId`를 기준으로 데이터를 받아 렌더링하도록 변경한다.

현재의 `boardVariant`는 “보드 디자인” 또는 “프레임 타입”에 가까우므로, 실제 보드 식별자인 `boardId`와 분리해서 관리한다.

## 카테고리 선택 페이지 UI

롤링페이퍼 인트로에서 `롤링페이퍼 메시지 남기러 가기`를 누르면 바로 작성 모달을 열지 않고, 먼저 카테고리 선택 페이지로 이동한다.

이 페이지는 사용자가 “무엇을 써야 하는지”를 빠르게 이해하도록 만드는 중간 단계다.

참고 UI는 운세 선택 화면처럼 구성할 수 있다.

```txt
상단: 뒤로가기 / 안내
중앙: 질문형 헤드라인
하단: 카테고리 카드 리스트
```

예시 문구:

```txt
오늘의 축제에서
어떤 순간을 남기고 싶나요?
```

또는:

```txt
경북대학교 80주년에
남기고 싶은 이야기를 골라주세요
```

카테고리 카드는 운세 카드처럼 명확한 아이콘과 짧은 라벨을 가진다.

예시:

```ts
const rollingPaperCategoryCards = [
  {
    id: 1,
    icon: '😊',
    title: '오늘의 기분',
    description: '지금 축제를 즐기는 마음을 남겨요',
  },
  {
    id: 2,
    icon: '🍢',
    title: '주막 음식',
    description: '가장 맛있었던 메뉴를 공유해요',
  },
  {
    id: 3,
    icon: '🎤',
    title: '공연 후기',
    description: '기억에 남는 무대를 기록해요',
  },
  {
    id: 4,
    icon: '📸',
    title: '오늘의 순간',
    description: '가장 기억에 남는 장면을 남겨요',
  },
  {
    id: 5,
    icon: '🎂',
    title: '80주년 축하',
    description: '경북대학교 80주년 축하 메시지를 남겨요',
  },
];
```

### 카테고리 선택 후 흐름

카테고리를 선택하면 프론트는 해당 카테고리의 여유 있는 보드를 요청한다.

```txt
카테고리 카드 클릭
-> GET /rolling-paper/categories/:categoryId/available-board
-> 여유 있는 boardId 반환
-> 해당 보드 화면으로 이동
-> 작성 모달 열기 또는 메시지 남기기 버튼 노출
```

사용자가 카테고리를 선택한 뒤에는 다시 질문을 고민하지 않고 바로 작성 흐름으로 들어가야 한다.

권장 흐름은 다음과 같다.

```txt
카테고리 선택
-> 여유 있는 보드 진입
-> 메시지 작성 모달 자동 오픈
```

다만 보드를 먼저 구경하고 작성하게 하고 싶다면, 자동 모달 오픈 대신 보드 화면의 `메시지 남기기` 버튼을 유지할 수 있다.

### UI 의도

카테고리 선택 페이지의 목적은 카테고리를 많이 보여주는 것이 아니라, 사용자가 작성할 메시지의 맥락을 빠르게 고르게 하는 것이다.

따라서 카테고리 수는 너무 많지 않아야 한다. 1차 구현에서는 `5개 내외`가 적당하다.

카드 UI는 다음 조건을 만족해야 한다.

- 모바일 한 화면에서 주요 카테고리가 바로 보여야 한다.
- 라벨은 짧고 직관적이어야 한다.
- 아이콘은 카테고리 의미를 즉시 이해할 수 있어야 한다.
- 선택 후 작성 흐름까지 이동이 짧아야 한다.
- 카테고리 선택이 또 다른 탐색 부담이 되면 안 된다.

### 라우팅 예시

```txt
/rolling-paper
-> 롤링페이퍼 소개 페이지

/rolling-paper/categories
-> 카테고리 선택 페이지

/rolling-paper/categories/:categoryId/boards/:boardId
-> 선택된 카테고리의 보드 페이지
```

또는 현재 라우팅을 유지한다면:

```txt
/rolling-paper/intro
/rolling-paper/category
/rolling-paper/board?categoryId=1&boardId=10
```

## 남은 의사결정

- 카테고리를 몇 개로 둘 것인가
- 카테고리별 보드를 몇 개씩 미리 만들 것인가
- 사용자가 보드를 직접 선택할 수 있게 할 것인가, 자동 배정할 것인가
- 카테고리 선택 후 작성 모달을 자동으로 열 것인가, 보드 화면에서 버튼을 누르게 할 것인가
- 보드가 가득 찬 경우 다음 여유 보드로 자동 이동할 것인가, 사용자에게 안내할 것인가
- 카테고리별 질문 문구를 축제 당일에도 변경할 수 있게 할 것인가
- 카테고리별로 보드 디자인을 다르게 둘 것인가

## 결론

롤링페이퍼는 단순히 많은 포스트잇을 수용하는 기능이 아니라, 사용자가 축제 경험을 쉽게 남기도록 돕는 기능이다.

따라서 대규모 참여를 고려할 때는 “큰 보드 하나를 탐색하는 경험”보다 “질문 카테고리로 작성 맥락을 제공하고, 여러 보드로 수용량을 분산하는 구조”가 더 적합하다.

최종적으로는 질문 카테고리 기반 롤링페이퍼를 중심으로 구현 방향을 변경하는 것이 합리적이다.
