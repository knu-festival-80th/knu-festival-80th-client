# Rolling Paper Board Layout

## 목표

- 모바일에서 손가락으로 확대/축소하고, 확대된 상태에서도 자연스럽게 위치를 확인할 수 있게 한다.
- 포스트잇은 자유 배치를 유지하되, 중앙 프레임 이미지 주변에는 붙지 않게 한다.
- 한 보드에서 최대 `100개`까지 배치할 수 있게 한다.
- 작성 모달과 실제 보드 화면이 같은 좌표계와 같은 줌 규칙을 사용하게 한다.

## 좌표계

롤링페이퍼는 화면 크기와 분리된 `흰색 논리 캔버스`를 기준으로 동작한다.

- 논리 캔버스: `852 x 852`
- 실제 보드 뷰포트: `100% x 509` (`DefaultLayout` 안에서 최대 `600 x 509`)
- 작성 모달 미리보기 뷰포트: `287 x 375`
- 저장 좌표: 포스트잇 중심점의 `x`, `y` 퍼센트

이 구조로 바꾼 이유는 두 가지다.

- 확대/축소가 들어가면 DOM 실제 픽셀보다 논리 좌표가 훨씬 안정적이다.
- 보드 화면과 작성 모달이 다른 크기여도 같은 위치 계산을 그대로 공유할 수 있다.

## 핵심 상수

| 항목             | 값                | 이유                                                                                   |
| ---------------- | ----------------- | -------------------------------------------------------------------------------------- |
| 논리 캔버스      | `852 x 852`       | Figma `2099:4601` 도면과 같은 정사각 좌표계를 사용한다.                                |
| 보드 뷰포트      | `100% x 509`      | 기존 모바일 페이지 레이아웃은 유지하되, `600px` 프레임에서는 양옆까지 꽉 채운다.       |
| 작성 미리보기    | `287 x 375`       | 모달 내부에서 손가락 조작과 버튼 영역을 함께 확보한다.                                 |
| 프레임 크기      | `320 x 320`       | Figma 도면의 중앙 마스코트 프레임 크기와 맞춘다.                                       |
| 프레임 금지 여백 | `26px`            | 이미지에 닿아 보이지 않도록 최소 여백을 둔다.                                          |
| 포스트잇 폭      | `80px`            | 보드에 붙은 메시지가 최소한 식별될 수 있게 키운 기준값이다.                            |
| 충돌 박스        | 렌더 크기의 `60%` | 100개 배치를 고려하되, 텍스트 영역이 과하게 겹치지 않도록 핵심 영역을 충돌로 판정한다. |
| 확대 범위        | `0.7 ~ 12`        | 축소 전체 보기, 수동 확대, 포스트잇 포커스 확대를 모두 지원한다.                       |

## 중앙 프레임과 금지 영역

중앙 이미지는 `320 x 320` 크기로 배치하고, 주변 `26px`은 추가 금지 영역으로 둔다.

- 프레임 실제 크기: `320 x 320`
- 금지 여백: `26px`
- 금지 판정: 포스트잇 충돌 박스와 프레임 금지 사각형이 겹치면 배치 불가

픽셀 마스크가 아니라 사각형 금지 영역을 쓰는 이유는 다음과 같다.

- 계산 비용이 작다.
- 작성 모달과 실제 보드에서 같은 판정을 재사용할 수 있다.
- 사용자가 보기에 "사진 주변에는 안 붙는다"는 규칙이 일관되다.

## 마스코트 중앙 정렬

Figma `2099:4601` 도면은 `852 x 852` 정사각 캔버스 안에서 중앙 프레임을 `320 x 320`으로 둔다.

현재 프론트도 같은 기준을 사용해서 프레임 좌상단을 `(266, 266)`에 배치한다.

- 목적: Figma 도면과 같은 크기감으로 중앙 마스코트를 보여주기 위함
- 결과: 작성 모달과 실제 보드 모두 같은 프레임 위치와 금지 영역을 사용한다

## 포스트잇 배치 규칙

- 포스트잇 폭: `80px`
- 포스트잇은 색상별 원본 비율을 유지한다.
- 충돌 판정은 전체 외곽이 아니라 `60%` 크기의 내부 박스를 사용한다.
- 사용자가 찍은 위치가 막혀 있으면, 가장 가까운 빈 위치를 원형 탐색으로 찾는다.
- 보드당 최대 개수는 `100개`다.

충돌 박스를 실제 렌더보다 조금 작게 둔 이유는 포스트잇 SVG 바깥에 여백이 있기 때문이다. 시각적으로는 촘촘하게 붙이되, 텍스트가 서로 먹히는 상황은 줄이는 쪽이 UX가 더 낫다.

## 포스트잇 텍스트 렌더링

포스트잇 SVG는 색상별로 모양과 비율이 다르다. 그래서 텍스트를 단순히 스티커 중앙에 놓으면 일부 스티커에서는 텍스트가 그림 영역을 벗어나거나, 작성 중 브라우저 보정 UI가 실제 텍스트와 다른 위치에 표시될 수 있다.

현재는 `rollingPaperStickerText.ts`에서 색상별 텍스트 안전 영역을 별도로 관리한다.

```ts
type RollingPaperStickerTextConfig = {
  centerX: string;
  centerY: string;
  width: string;
  height: string;
  aspectRatio: number;
  heightRatio: number;
  maxLines: number;
  charsPerLine: number;
};
```

- `centerX`, `centerY`: 스티커 내부 텍스트 박스의 중심 위치
- `width`, `height`: 스티커 내부에서 텍스트가 들어갈 수 있는 안전 영역
- `aspectRatio`, `heightRatio`: 작성 textarea를 세로 중앙에 맞추기 위한 높이 계산값
- `maxLines`: 해당 안전 영역에서 허용하는 최대 줄 수
- `charsPerLine`: 줄 수를 예측하기 위한 줄당 기준 글자 수

표시 텍스트는 `RollingPaperSticker` 내부에서 중앙 정렬로 렌더링한다. 보드에 붙은 작은 포스트잇은 컨테이너 폭 기준 `cqw` 단위를 사용해서 보드 확대/축소와 함께 자연스럽게 커진다.

작성 중에는 실제 `textarea`를 같은 안전 영역 위에 올리고, `paddingTop`/`paddingBottom`을 계산해 텍스트 블록을 세로 중앙에 맞춘다. 이 방식으로 브라우저 커서와 드래그 선택 위치가 실제 텍스트 위치와 어긋나는 문제를 줄인다.

### 80자 텍스트 처리

현재 작성 제한은 `ROLLING_PAPER_MAX_MESSAGE_LENGTH = 80`이다. 텍스트는 작성 모달에서 `11px`, `line-height: 1.5`를 기준으로 고정하고, 길이에 따라 폰트 크기를 계속 줄이지 않는다.

- 입력 단계에서 `maxLength`로 80자를 1차 제한한다.
- 스티커별 `maxLines`와 `charsPerLine`을 기준으로 안전 영역을 넘는 입력은 추가로 제한한다.
- 보드 표시 텍스트는 `clamp(1.79px, 3.57cqw, 12px)`로 렌더링한다.

`line-clamp`는 사용하지 않는다. line-clamp는 영역을 넘어가면 `...`를 만들기 때문에 입력한 내용 일부가 화면에 보이지 않는 문제가 생긴다. 대신 입력 단계에서 안전 영역 기준을 넘는 문자를 반영하지 않는다.

작성 textarea에는 다음 처리를 적용한다.

- `spellCheck={false}`로 브라우저 맞춤법 밑줄을 제거한다.
- `autoCorrect="off"`와 `autoCapitalize="off"`로 모바일 자동 보정을 끈다.
- Grammarly 관련 `data-*` 속성을 꺼서 외부 문법 도구 UI가 보이지 않게 한다.
- 커서와 드래그 selection 색상은 `secondary-blue` 계열로 맞춘다.

이 구조에서 백엔드는 텍스트 자체와 `colorId`만 저장하면 된다. 텍스트 박스 위치, 줄 수, 글자 크기 보정은 프론트 렌더링 책임으로 둔다.

## 배치 흐름

작성 모달에서 사용자가 위치를 선택하면 다음 순서로 좌표가 결정된다.

1. 터치 좌표를 작성 미리보기 뷰포트 기준으로 읽는다.
2. 현재 확대 배율과 팬 위치를 반영해서 논리 캔버스 좌표로 변환한다.
3. 논리 캔버스 좌표를 `x`, `y` 퍼센트로 바꾼다.
4. 포스트잇이 캔버스 밖으로 나가지 않도록 1차 보정한다.
5. 중앙 프레임 금지 영역과 기존 포스트잇 충돌을 검사한다.
6. 충돌하면 요청 위치 주변을 원형으로 탐색해서 가장 가까운 빈 위치를 찾는다.
7. 최종 위치를 퍼센트 좌표로 저장한다.

이 흐름을 보드 화면과 작성 모달이 같은 헬퍼 함수로 공유한다. 그래서 미리보기에서 보인 위치와 실제 보드에 붙는 위치가 달라지는 문제를 줄일 수 있다.

## 확대/축소와 팬

- 지원 배율: `0.7 ~ 12`
- 기본 배율: `1.0`
- 배율 스텝 버튼: `0.5`
- 포스트잇 포커스 배율: 최대 `10.5`
- 모바일 제스처:
  - 작성 모달: 한 손가락으로 배치, 두 손가락으로 확대/축소와 화면 이동
  - 보드 화면: 포스트잇 터치 시 해당 포스트잇으로 확대, 두 손가락으로 확대/축소와 화면 이동

기본 배율은 `1.0`으로 유지하지만, 최소 배율은 `0.7`까지 허용한다. `600px` 프레임에서는 기본 상태가 양옆을 꽉 채우는 cover 화면이고, 사용자가 축소 버튼이나 핀치 제스처로 더 넓은 도면을 확인할 수 있어야 하기 때문이다.

최대 배율 `12`는 사용자가 직접 확대해서 작은 포스트잇을 읽을 수 있게 하기 위한 상한이다. 포스트잇을 터치했을 때는 기본적으로 `10.5`까지 확대하되, 세로로 긴 스티커는 뷰포트 여백을 확보할 수 있도록 실제 포커스 배율을 낮춘다.

- 개별 포스트잇 메시지를 읽기 쉽다.
- 긴 스티커가 화면 밖으로 과하게 벗어나지 않는다.
- 원점 버튼을 누르면 기본 배율과 팬 위치, 포커스 상태를 함께 초기화한다.

보드 페이지는 상위 `DefaultLayout`의 기존 모바일 프레임을 유지한다. 대신 보드 뷰포트는 고정 `375px`이 아니라 컨테이너 폭 `100%`를 사용한다. 모바일에서는 자연스럽게 약 `375 x 509`로 보이고, 데스크톱의 `600px` 프레임에서는 `600 x 509`까지 커져 양옆을 채운다.

내부 흰색 도면은 Figma 기준 `852 x 852` 정사각 캔버스이고, 실제 보드 뷰포트 크기를 `ResizeObserver`로 측정한 뒤 `cover-scale`로 렌더링한다. 기본 상태에서는 보드 영역을 꽉 채우고, 축소하면 더 넓은 도면을 한 번에 볼 수 있다.

작성 모달은 작은 영역 안에서 배치 가능한 전체 범위를 한 번에 보여주는 것이 더 중요하므로 기존 `287 x 375` 고정 뷰포트와 `contain-scale`을 유지한다.

## 제스처 규칙

작성 모달과 보드 화면은 목적이 다르기 때문에 한 손가락 동작만 다르게 처리한다.

| 화면      | 한 손가락           | 두 손가락            |
| --------- | ------------------- | -------------------- |
| 작성 모달 | 포스트잇 위치 선택  | 확대/축소 및 팬 이동 |
| 보드 화면 | 확대된 보드 팬 이동 | 확대/축소 및 팬 이동 |

보드 화면에서 한 손가락 팬을 허용한 이유는 읽기 화면에서는 포스트잇을 새로 배치하지 않기 때문이다. 반대로 작성 모달에서는 한 손가락을 위치 선택에 써야 하므로, 화면 이동은 두 손가락 제스처에 묶었다.

## 수정 시 주의사항

- `ROLLING_PAPER_CANVAS_DIMENSIONS`는 Figma 도면 기준 `852 x 852`와 맞춰야 한다.
- 보드 화면의 팬/줌 계산은 고정 상수가 아니라 실제 DOM 뷰포트 크기를 기준으로 해야 한다.
- 보드 화면은 `cover-scale`, 작성 모달은 `contain-scale`을 사용한다. 두 모드를 섞으면 팬 제한과 좌표 변환도 같은 모드로 계산해야 한다.
- `ROLLING_PAPER_CANVAS_DIMENSIONS`를 줄이면 `100개` 배치 테스트가 깨질 수 있다.
- `ROLLING_PAPER_FRAME_DIMENSIONS.blockedPadding`을 키우면 중앙 금지 영역이 넓어져 수용량이 줄어든다.
- `ROLLING_PAPER_NOTE_WIDTH`를 키우면 가독성은 좋아지지만 최대 수용량이 줄어든다.
- `COLLISION_SCALE`을 키우면 겹침 방지는 강해지지만 배치 가능한 개수가 줄어든다.
- 프레임 이미지별 시각 중심이 다르면 `FRAME_VARIANT_OFFSETS`로 보정해야 한다.

## 백엔드 공유 요약

현재 프론트는 논리 캔버스 기준 퍼센트 좌표를 사용한다. 백엔드도 같은 좌표계와 배치 상수를 공유해야 실제 저장 위치와 프론트 미리보기 위치가 어긋나지 않는다.

### 저장 데이터 구조

```ts
type RollingPaperNote = {
  id: string;
  boardId: string;
  boardVariant: number;
  message: string;
  colorId: 'red' | 'yellow' | 'green' | 'blue' | 'purple' | 'pink';
  x: number;
  y: number;
  requestedX?: number;
  requestedY?: number;
  createdAt: string;
};
```

- `x`, `y`: 최종 확정된 포스트잇 중심 좌표. `0 ~ 100` 퍼센트 값으로 저장한다.
- `requestedX`, `requestedY`: 사용자가 처음 요청한 좌표. 자동 보정 여부를 디버깅할 때만 필요하다.
- `boardVariant`: 중앙 프레임 종류와 시각 중심 보정값을 결정한다.
- `colorId`: 스티커별 비율과 충돌 박스 계산에 필요하다.

### API 책임

`POST /rolling-paper/boards/:boardId/notes`는 프론트 요청 좌표를 그대로 저장하지 않고, 백엔드에서 최종 좌표를 확정해야 한다.

```ts
type CreateRollingPaperNoteRequest = {
  message: string;
  colorId: RollingPaperNote['colorId'];
  requestedX: number;
  requestedY: number;
  idempotencyKey: string;
};

type CreateRollingPaperNoteResponse = {
  note: RollingPaperNote;
  adjusted: boolean;
  adjustmentDistancePx: number;
};
```

백엔드 처리 순서는 다음 정도면 충분하다.

1. `requestedX`, `requestedY`를 `0 ~ 100` 범위로 보정한다.
2. `boardId` 기준으로 현재 포스트잇 개수와 기존 좌표를 조회한다.
3. 보드가 `100개` 이상이면 저장하지 않는다.
4. 중앙 프레임 금지 영역과 기존 포스트잇 충돌을 검사한다.
5. 충돌하면 가까운 빈 위치를 탐색한다.
6. 최종 `x`, `y`를 저장하고 응답한다.

### 구현 주의사항

- 배치 판정의 최종 권위는 백엔드가 가져야 한다. 프론트 계산은 미리보기 용도로만 본다.
- 같은 보드에 동시에 저장 요청이 들어오면 트랜잭션 안에서 보드 단위 락을 짧게 잡는다.
- 중복 클릭과 네트워크 재시도를 대비해 `idempotencyKey`를 받는다.
- 스티커별 `hitboxDiff`나 충돌 박스 상수는 프론트와 백엔드가 같은 값으로 관리해야 한다.
- 요청 위치에서 너무 먼 곳으로 보정될 수 있으면 최대 보정 반경을 두고 실패 응답을 반환하는 편이 UX상 낫다.
- DB에는 퍼센트 좌표를 저장하고, 화면 픽셀 좌표는 프론트 렌더링 시점에만 계산한다.

## 간단 데모

프론트와 백엔드 배치 판정을 함께 확인하는 독립 데모를 `demos/rolling-paper-placement-demo.mjs`에 둔다.

```bash
pnpm demo:rolling-paper
```

실행 후 `http://localhost:4174`에서 확인한다.

데모 서버는 Node 기본 `http` 모듈만 사용한다. 별도 DB는 없고 메모리에 포스트잇을 저장한다.

- `GET /`: 데모 웹 페이지
- `GET /api/board`: 현재 보드 상태 조회
- `POST /api/notes`: 요청 좌표를 받아 백엔드에서 최종 좌표 확정
- `POST /api/reset`: 메모리 보드 초기화

`POST /api/notes`는 다음 흐름을 보여준다.

1. 프론트가 사용자가 드롭한 `requestedX`, `requestedY`를 보낸다.
2. 백엔드가 스티커별 `hitboxDiff`를 적용해 충돌 박스를 만든다.
3. 중앙 프레임 금지 영역과 기존 포스트잇 충돌을 검사한다.
4. 겹치면 가까운 빈 좌표를 탐색한다.
5. 백엔드가 확정한 최종 `x`, `y`를 응답한다.
6. 프론트는 요청 좌표와 최종 좌표 차이를 서버 로그 패널에 표시한다.

데모에는 `withBoardLock` 큐가 들어 있다. 실제 DB 락은 아니지만, 같은 보드에 동시에 들어온 요청을 서버 내부에서 순서대로 처리하는 흐름을 보여준다. 실서비스에서는 이 부분을 DB 트랜잭션과 보드 단위 advisory lock 또는 row lock으로 바꾸면 된다.

## 검증 포인트

- 중앙 프레임 위와 주변 여백에는 포스트잇이 배치되지 않는다.
- 메인 마스코트 프레임은 Figma 기준 `(266, 266, 320, 320)` 위치에 보인다.
- `100개`까지 연속 배치가 가능하다.
- 보드 화면과 작성 모달의 확대/축소 범위가 동일하다.
- 확대 후 팬 위치는 항상 캔버스 범위 안으로 자동 보정된다.
