# 롤링페이퍼 목적성 및 UX 방향

## 1. 핵심 목적

롤링페이퍼는 메시지 관리 도구가 아니라 `축제 참여의 흔적을 내가 원하는 자리에 직접 붙이는 감성 장치`입니다.

사용자는 이 기능을 통해 다음 감정을 얻어야 합니다.

- 내가 지금 축제에 참여하고 있다는 느낌
- 내 메시지가 모두의 롤링페이퍼에 실제로 붙었다는 즉각적인 피드백
- 내가 직접 자리를 골라 붙였다는 소유감
- 다른 사람들도 같은 축제를 즐기고 있다는 현장감
- 오래 머무르지 않아도 가볍게 읽고 빠져나갈 수 있는 부담 없는 경험

따라서 이 기능의 중심은 "내가 쓴 글을 나중에 다시 찾기"나 "모든 글을 꼼꼼히 탐색하기"가 아닙니다.

## 2. 제품 한 문장

`축제의 순간을 30초 안에 포스트잇으로 남기고, 내가 고른 위치에 직접 붙이며 모두의 롤링페이퍼를 함께 완성하는 기능`

## 3. 사용자 행동 기준

### Primary Job

사용자는 축제 현장에서 짧은 시간 안에 메시지를 남기고, 빈 공간을 골라 직접 붙인 뒤 결과를 확인합니다.

성공 조건:

- 30초 안에 작성 완료 가능
- 작성 후 내 포스트잇 위치를 즉시 확인
- 직접 배치하되 기존 포스트잇을 덮거나 밀지 않음
- 별도 학습 없이 사용 가능
- 모바일 한 손 조작 가능

### Secondary Job

사용자는 다른 사람들의 메시지를 가볍게 훑으며 축제 분위기를 느낍니다.

성공 조건:

- 처음 보는 사람도 바로 읽을 수 있음
- 확대/축소 조작 없이도 핵심 메시지가 보임
- 랜덤/최신/추천 묶음으로 부담 없이 탐색 가능

### Tertiary Job

일부 사용자는 큰 보드를 확대/축소하며 더 깊게 탐색할 수 있습니다.

성공 조건:

- 탐험 모드는 선택 사항
- 기본 사용 흐름을 방해하지 않음
- 기술적 재미는 있지만 필수 조작은 아님

## 4. 하지 않을 것

MVP에서는 다음을 핵심으로 두지 않습니다.

- 내가 쓴 포스트잇을 나중에 다시 찾기
- 닉네임 검색
- 찾기 코드
- 수정/삭제
- 모든 메시지를 완독하게 만드는 탐색
- 복잡한 무한 캔버스 조작
- 기존 사용자의 포스트잇을 밀어내거나 재배치하는 기능
- 다른 사람의 포스트잇을 가리는 배치

## 5. 권장 UX 구조

### 5.1 기본 모드: 롤링페이퍼 읽기

처음 진입하면 줌 보드가 아니라 `읽기 쉬운 롤링페이퍼 뷰`를 보여줍니다.

구성:

- 상단: 총 메시지 수, 오늘 작성된 메시지 수
- 중앙: 12~20개의 포스트잇을 보기 좋은 밀도로 배치
- 하단: `메시지 남기기`, `랜덤으로 보기`, `전체 보드 탐험`

이 모드는 사용자가 아무 조작 없이도 메시지를 읽을 수 있어야 합니다.

### 5.2 작성 및 직접 배치 플로우

작성은 바텀시트로 처리합니다.

입력:

- 닉네임
- 메시지
- 포스트잇 색상

작성 후:

1. 작성한 포스트잇 미리보기를 보여줍니다.
2. 보드에서 붙일 위치를 선택합니다.
3. 선택 위치가 비어 있으면 `이 위치에 붙이기`를 활성화합니다.
4. 선택 위치가 기존 포스트잇과 충돌하면 가까운 빈 위치를 추천합니다.
5. 붙이기 완료 후 해당 위치를 화면 중앙에 두고 2초 정도 하이라이트합니다.
6. 이후 사용자는 그대로 둘러보거나 페이지를 나갈 수 있습니다.

핵심은 `직접 붙였다는 감각`과 `작성 완료 후 위치 확인`입니다. 다음 방문에서 다시 찾는 기능은 제공하지 않습니다.

### 5.3 탐험 모드: 줌 보드

줌/팬 보드는 기본 화면이 아니라 선택형 고급 탐색입니다.

진입 버튼:

- `전체 보드 탐험`
- `100개의 포스트잇 보기`

탐험 모드 역할:

- 포트폴리오 기술 과제
- 대형 롤링페이퍼 감성 연출
- 축제 종료 후 아카이브성 탐색
- 데스크톱 또는 태블릿에서 더 좋은 경험 제공

모바일에서는 다음 보조 장치가 필요합니다.

- 확대/축소 버튼
- 원점 복귀
- 미니맵
- 현재 위치 표시
- 길을 잃었을 때 돌아오는 CTA

## 6. 직접 배치와 100개 줌 보드에 대한 판단

직접 배치형 롤링페이퍼는 목적에 잘 맞습니다. 롤링페이퍼는 정렬된 게시판보다 서로 다른 사람들이 자유롭게 남긴 흔적에 가깝기 때문입니다.

다만 모든 것을 완전 자유 배치로 열면 기존 포스트잇을 가리거나, 빈 공간을 찾기 어렵거나, 모바일에서 조작이 피곤해질 수 있습니다. 따라서 `보호된 자유 배치`가 적합합니다.

보호 규칙:

- 기존 포스트잇은 움직이지 않습니다.
- 새 포스트잇은 기존 포스트잇의 본문 영역을 덮을 수 없습니다.
- 약간의 시각적 겹침은 허용하되, 충돌 기준을 넘으면 배치 불가로 처리합니다.
- 사용자가 고른 위치 주변의 가장 가까운 빈 공간을 추천합니다.
- 동시 작성으로 충돌이 생기면 서버가 가장 가까운 유효 위치를 다시 제안합니다.

100개 줌 보드는 기술적으로 좋은 도전이지만, 모바일 기본 UX로 두면 사용자 친화적이지 않을 수 있습니다.

따라서 다음처럼 역할을 나눕니다.

| 구분            | 기본 읽기 모드              | 탐험 모드            |
| --------------- | --------------------------- | -------------------- |
| 목적            | 빠르게 작성하고 직접 붙이기 | 깊게 둘러보기        |
| 조작            | 스크롤, 탭, 위치 선택       | pan, zoom, pinch     |
| 포스트잇 수     | 12~20개 노출                | 보드당 최대 100개    |
| 난도            | 낮음                        | 높음                 |
| 대상            | 모든 사용자                 | 관심 있는 사용자     |
| 포트폴리오 가치 | UX 설계                     | 렌더링/제스처 최적화 |

결론:

- 기본 UX는 단순해야 합니다.
- 기술적 도전은 탐험 모드에 숨깁니다.
- 작성 직후에는 선택 위치로 카메라 이동을 짧게 사용해 "내가 붙였다"는 감각을 전달합니다.

## 7. 프론트엔드 포트폴리오 방향

포트폴리오에서 강조할 문제는 "복잡한 기능을 많이 넣었다"가 아니라 다음이어야 합니다.

`사용자에게는 단순한 방명록처럼 보이지만, 내부적으로는 대량 포스트잇 데이터를 보드 단위로 나누고, 필요한 순간에만 줌/팬 탐색과 렌더링 최적화를 제공했다.`

핵심 기술 과제:

- 기본 모드와 탐험 모드의 상태 공유
- 사용자가 선택한 위치의 충돌 검사
- 가장 가까운 빈 위치 추천
- 작성 후 포스트잇 위치로 카메라 이동
- 100개 보드의 좌표 기반 렌더링
- 기존 포스트잇을 보호하는 spatial index
- zoom level별 LOD
- viewport culling
- 모바일 pointer gesture
- 미니맵
- React Query prefetch
- optimistic UI
- 1만 개 mock 데이터 성능 검증

## 8. 최종 권장안

롤링페이퍼의 목적은 `축제 참여의 즉각적인 흔적 남기기`입니다.

따라서 MVP의 기본 흐름은 다음이 가장 적합합니다.

1. 짧게 메시지를 작성한다.
2. 빈 공간을 골라 직접 붙인다.
3. 내가 쓴 포스트잇이 붙는 순간을 본다.
4. 다른 사람의 메시지를 가볍게 훑는다.
5. 원하면 전체 보드 탐험 모드로 들어간다.

직접 배치는 롤링페이퍼의 목적에 포함합니다. 100개 줌 보드는 기본 사용 흐름 전체가 아니라 `직접 붙일 위치를 고르는 순간`과 `선택형 탐험 모드`에서 사용합니다. 이렇게 해야 사용자 친화성과 프론트엔드 기술적 도전을 동시에 가져갈 수 있습니다.
