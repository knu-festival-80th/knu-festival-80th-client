# 롤링페이퍼 방명록 기능 기획 명세서

롤링페이퍼의 제품 목적과 UX 방향은 [GUESTBOOK_PURPOSE_AND_UX_DIRECTION.md](./GUESTBOOK_PURPOSE_AND_UX_DIRECTION.md)를 우선 기준으로 삼습니다. 줌/팬 보드와 프론트엔드 포트폴리오 관점의 상세 기획은 [GUESTBOOK_SEARCH_AND_PORTFOLIO_SPEC.md](./GUESTBOOK_SEARCH_AND_PORTFOLIO_SPEC.md)를 함께 참고합니다.

## 1. 개발 가능성 판단

구현 가능합니다. 다만 롤링페이퍼의 목적은 복잡한 캔버스 도구를 제공하는 것이 아니라, 축제 참여자가 짧은 시간 안에 메시지를 남기고 "내가 고른 자리에 포스트잇을 직접 붙였다"는 감각을 받게 하는 것입니다.

따라서 기본 UX는 단순해야 합니다. 100개 단위 줌 보드는 메인 경험이 아니라 `전체 보드 탐험 모드`로 분리합니다. 사용자는 기본 화면에서 쉽게 작성하고 읽을 수 있고, 더 깊게 보고 싶은 경우에만 줌/팬 보드로 진입합니다.

이 방식의 장점:

- 일반 사용자는 줌/팬을 몰라도 메시지를 작성하고 읽을 수 있습니다.
- 작성 후 빈 공간을 직접 고르고, 붙인 위치로 이동해 즉각적인 피드백을 줍니다.
- 기존 포스트잇을 덮거나 밀지 않는 보호 배치 규칙을 적용할 수 있습니다.
- 탐험 모드에서는 100개 포스트잇 보드, 미니맵, LOD, culling 등 기술 과제를 다룰 수 있습니다.
- 사용자 친화성과 포트폴리오 기술 난도를 분리할 수 있습니다.

## 2. 권장 콘셉트

### 기능명

80주년 대동제 롤링페이퍼

### 핵심 문장

경북대 80주년 대동제에 남기는 모두의 한마디를 내가 원하는 빈자리에 포스트잇처럼 붙이고, 축제의 현장감을 함께 느끼는 방명록입니다.

### 권장 방식

- 기본 화면은 12~20개의 포스트잇을 읽기 좋은 밀도로 보여줍니다.
- 사용자는 바로 메시지를 남길 수 있습니다.
- 작성 후 사용자는 보드에서 비어 있는 위치를 직접 선택합니다.
- 선택 위치가 기존 포스트잇과 충돌하면 가까운 빈 위치를 추천합니다.
- 붙이기 완료 후 사용자는 자신이 남긴 포스트잇 위치를 즉시 확인합니다.
- 100개 단위 줌 보드는 `전체 보드 탐험 모드`로 제공합니다.
- 다음 방문에서 내 글을 다시 찾는 기능은 MVP에서 제공하지 않습니다.

## 3. 사용자 목표

- 축제 방문자가 짧은 응원, 추억, 감상을 남길 수 있어야 합니다.
- 작성 후 본인이 고른 위치에 포스트잇이 실제 보드에 붙은 느낌을 받아야 합니다.
- 다른 사람의 메시지를 부담 없이 훑으며 축제 분위기를 느낄 수 있어야 합니다.
- 모바일 현장에서 빠르게 작성하고 빠르게 닫을 수 있어야 합니다.

## 4. MVP 범위

### 포함

- 방명록 메인 진입 화면
- 읽기 쉬운 기본 롤링페이퍼 화면
- 메시지 작성
- 작성 후 포스트잇 위치 직접 선택
- 선택 위치 충돌 검사
- 가까운 빈 위치 추천
- 붙이기 완료 후 포스트잇 위치로 자동 이동
- 작성 포스트잇 하이라이트
- 전체 보드 탐험 모드
- 탐험 모드에서 줌/팬 가능한 100개 포스트잇 보드
- 탐험 모드에서 미니맵 또는 현재 위치 인디케이터
- 메시지 신고
- 운영자 숨김 처리를 고려한 상태값
- 로딩, 빈 상태, 오류 상태

### 제외

- 로그인
- 닉네임 검색
- 내 메시지 목록
- 찾기 코드
- 메시지 수정/삭제
- 실시간 동시 편집
- 기존 포스트잇을 밀어내는 배치
- 다른 포스트잇의 본문을 가리는 배치
- 여러 사용자가 동시에 같은 보드를 편집하는 실시간 협업 커서
- 무한 캔버스
- 이미지 첨부
- 댓글, 좋아요, 대댓글

## 6. 화면 명세

### 6.1 방명록 메인

목적: 사용자가 기능을 이해하고 바로 작성하거나 보드를 둘러볼 수 있게 합니다.

필수 요소:

- 제목: `80주년 대동제 롤링페이퍼`
- 보조 문구: `축제의 순간을 포스트잇으로 남겨보세요.`
- 주요 버튼: `메시지 남기기`
- 보조 버튼: `보드 둘러보기`
- 상태 요약: 총 메시지 수, 현재 보드 수
- 최신 보드 미리보기

디자인 메모:

- 설명형 랜딩 페이지보다 바로 작성할 수 있는 도구 화면에 가깝게 구성합니다.
- 모바일 첫 화면 안에 작성 버튼이 보여야 합니다.
- 축제 감성은 포스트잇 색, 종이 질감, 보드 배경, 스티커 느낌으로 표현합니다.

### 6.2 줌 보드

목적: 100개 포스트잇을 하나의 큰 보드에 배치하고, 사용자가 확대/축소하며 탐색할 수 있게 합니다.

필수 요소:

- 보드 번호: `Board 12 / 100`
- 이전/다음/랜덤 보드 버튼
- 줌 가능한 보드 viewport
- 확대/축소/원점 복귀 버튼
- 미니맵 또는 현재 위치 인디케이터
- 작성 CTA
- 신고 버튼

권장 인터랙션:

- 한 손가락 드래그: 보드 이동
- 두 손가락 pinch: 확대/축소
- 더블 탭: 확대 토글
- 버튼: 확대, 축소, 원점 복귀

포스트잇 카드 구성:

- 닉네임
- 메시지 본문
- 작성 시간 또는 축제 날짜
- 신고 버튼

카드 본문 길이:

- 최대 120자
- 보드에서는 3~5줄 말줄임
- 탭하면 상세 모달에서 전체 노출

### 6.3 메시지 작성 및 직접 배치

목적: 현장에서 빠르게 메시지를 작성하고, 사용자가 고른 빈 공간에 직접 붙이는 경험을 제공합니다.

권장 UI:

- 모바일에서는 바텀시트 작성 폼을 권장합니다.
- 작성 중에도 현재 보드 맥락이 유지되면 좋습니다.

필수 입력:

- 닉네임: 2~12자
- 메시지: 1~120자
- 포스트잇 색상 선택: 4~6개 프리셋

선택 입력:

- 소속/태그: 재학생, 졸업생, 방문객, 스태프 등

배치 단계:

- 작성 폼을 완료하면 포스트잇 미리보기를 보드 위에 표시합니다.
- 사용자는 보드의 원하는 위치를 탭하거나 드래그해 배치 후보를 정합니다.
- 배치 가능한 위치는 초록/파란 테두리, 불가능한 위치는 빨간 테두리로 즉시 피드백합니다.
- 기존 포스트잇과 충돌하면 `가까운 빈 위치로 이동` 액션을 제공합니다.
- 최종 붙이기 전에는 기존 포스트잇을 움직이거나 가리지 않습니다.

유효성:

- 공백만 입력 불가
- 금칙어 포함 시 등록 불가 또는 검수 대기
- 동일 기기 연속 작성 제한
- 선택 위치가 보드 경계를 벗어나면 등록 불가
- 선택 위치가 기존 포스트잇 본문 영역과 충돌하면 등록 불가

작성 완료:

- 완료 토스트: `롤링페이퍼에 메시지를 붙였어요.`
- 서버가 확정한 `boardNo`, `x`, `y` 좌표로 카메라 이동
- 새 포스트잇 2초 하이라이트
- 이후 내 글 찾기 기능은 제공하지 않음

### 6.4 신고 및 숨김

목적: 공개 방명록에서 부적절한 콘텐츠를 빠르게 줄입니다.

필수 요소:

- 카드별 신고 버튼
- 신고 사유 선택: 욕설/비방, 개인정보, 광고, 기타
- 신고 완료 후 해당 사용자에게 중복 신고 방지
- 운영자 숨김 상태의 메시지는 일반 사용자에게 보이지 않음

운영 정책:

- 신고 누적 기준 자동 임시 숨김을 고려합니다.
- 운영자 페이지는 MVP 이후 별도 어드민 기능으로 분리해도 됩니다.

## 8. API 초안

프론트엔드 구현 기준으로 필요한 API입니다.

```txt
GET /guestbook/summary
  총 메시지 수, 총 보드 수, 최신 보드 번호

GET /guestbook/boards/:boardNo
  특정 보드와 포스트잇 목록

GET /guestbook/boards/random
  랜덤 보드 번호 또는 보드 데이터

POST /guestbook/notes
  포스트잇 작성

POST /guestbook/notes/placement/validate
  선택 위치 배치 가능 여부 검사

POST /guestbook/notes/:noteId/report
  포스트잇 신고
```

작성 요청 예시:

```json
{
  "nickname": "북문지킴이",
  "message": "80주년 대동제 오래 기억할게요!",
  "color": "yellow",
  "tag": "student",
  "boardNo": 18,
  "x": 1240,
  "y": 860,
  "rotation": -3
}
```

작성 응답 예시:

```json
{
  "id": "note_123",
  "boardNo": 18,
  "x": 1240,
  "y": 860,
  "rotation": -3
}
```

배치 검사 요청 예시:

```json
{
  "boardNo": 18,
  "x": 1240,
  "y": 860,
  "width": 220,
  "height": 180,
  "rotation": -3
}
```

배치 검사 응답 예시:

```json
{
  "isValid": false,
  "reason": "overlap",
  "nearestValidPosition": {
    "x": 1320,
    "y": 910,
    "rotation": 2
  }
}
```

## 9. 보호된 자유 배치 규칙

권장 방식은 `사용자 직접 선택 + 서버 최종 검증`입니다.

- 사용자는 현재 보드에서 원하는 위치를 직접 선택합니다.
- 클라이언트는 현재 로드된 포스트잇 기준으로 즉시 충돌 여부를 미리 계산합니다.
- 서버는 작성 요청 시점에 같은 검사를 다시 수행해 최종 좌표를 확정합니다.
- 기존 포스트잇은 움직이지 않습니다.
- 새 포스트잇은 기존 포스트잇의 본문 영역을 덮을 수 없습니다.
- 약간의 시각적 겹침은 허용할 수 있지만, 최소 여백과 본문 가독성 기준을 넘으면 배치 불가입니다.
- 현재 보드가 100개로 가득 차면 새 보드를 생성하거나 다른 보드를 추천합니다.
- 숨김 처리된 메시지가 생겨도 좌표는 유지합니다.
- 동시 작성으로 충돌이 발생하면 서버는 `nearestValidPosition`을 반환하고, 사용자는 해당 위치로 다시 확인할 수 있습니다.

이 방식의 장점:

- 사용자가 실제 롤링페이퍼에 직접 붙였다는 감각을 얻습니다.
- 같은 보드를 다시 열어도 기존 포스트잇 위치가 바뀌지 않습니다.
- 이전 사용자의 메시지를 보호할 수 있습니다.
- 프론트엔드는 현재 보드 100개 이하만 렌더링합니다.
- 1만 개 메시지도 100개 보드로 나뉘어 처리됩니다.
- 신고/숨김 처리 후에도 레이아웃 재계산이 단순합니다.

## 10. 성능 기준

모바일 중심이므로 다음 기준을 권장합니다.

- 보드 진입 시 첫 API 응답 전 skeleton 표시
- 현재 보드와 인접 보드만 fetch
- CSS transform은 보드 레이어에만 적용
- pan/zoom은 `requestAnimationFrame`으로 업데이트
- 포스트잇 카드에는 이미지 첨부 금지
- 줌 레벨별 LOD 렌더링 적용
- viewport culling으로 화면 주변 포스트잇만 렌더링 가능하게 설계
- React Query로 보드 데이터 캐싱
- 이전/다음 보드 prefetch 적용
- 작성 완료 후 optimistic UI 또는 즉시 재조회

## 13. UX 의사결정

### 무한 캔버스

장점:

- 컨셉이 강합니다.
- 완성 시 시각적 임팩트가 큽니다.

단점:

- 모바일 성능 리스크가 큽니다.
- 사용자가 길을 잃기 쉽습니다.
- 전체 데이터 로딩과 좌표 관리가 복잡합니다.

판단: 부적합합니다.

### 100개 단위 줌 보드

장점:

- 큰 롤링페이퍼 감성이 살아납니다.
- 모바일 성능을 제어할 수 있습니다.
- 줌/팬, LOD, culling 등 기술 과제가 명확합니다.
- 붙이기 완료 후 위치 이동 UX를 강하게 만들 수 있습니다.
- 사용자가 직접 붙이는 롤링페이퍼 감각을 구현할 수 있습니다.
- 충돌 검사, 가까운 빈 위치 추천, 동시 작성 보정 등 기술 과제가 명확합니다.

단점:

- 일반 그리드보다 구현 난도가 높습니다.
- 제스처와 스크롤 충돌을 신중히 처리해야 합니다.
- 사용자가 현재 위치를 잃지 않도록 미니맵이 필요합니다.
- 위치 선택 단계가 길어지지 않도록 배치 후보와 추천 위치 피드백이 빨라야 합니다.

판단: MVP 확장 및 포트폴리오 방향으로 가장 적합합니다.

## 17. 최종 권장안

이번 서비스에는 `보호된 자유 배치가 가능한 100개 단위 줌 보드형 롤링페이퍼`가 가장 적합합니다.

사용자에게는 큰 롤링페이퍼를 직접 둘러보고 원하는 빈자리에 포스트잇을 붙이는 경험을 제공합니다. 기존 포스트잇은 움직이거나 가리지 않고, 작성 후에는 확정 위치로 이동해 즉각적인 피드백을 줍니다. 로그인, 닉네임 검색, 내 글 찾기는 MVP에서 제외해 UX를 단순하게 유지합니다.

프론트엔드 관점에서는 카메라 좌표계, 모바일 제스처, 배치 충돌 검사, 가까운 빈 위치 추천, 렌더링 최적화, LOD, optimistic UI를 구현할 수 있어 포트폴리오 기술 과제로도 충분합니다.
